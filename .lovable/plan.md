

# Fix: Wallet ID Mismatch Between JS and Database

## Problem
The wallet ID displayed in the app (generated by JavaScript) does not match the wallet ID computed by the database function. When a user searches for another user's wallet ID, the database can't find it because the hashing algorithms differ.

**Root cause — hash overflow handling differs:**
- **JS** (`walletId.ts`): Uses `(h << 5) - h` with `| 0` — signed 32-bit integer overflow (values can be negative)
- **SQL** (`generate_wallet_id_from_phone`): Uses `h * 31` with `& x'7FFFFFFF'` — clears the sign bit, keeping values always positive

This means the same phone number produces different wallet IDs in the UI vs the database, so lookups always fail.

## Fix
Update the **SQL function** to match the JS behavior exactly (signed 32-bit with `| 0` semantics). PostgreSQL integers are already 32-bit signed, so we just need to remove the `& x'7FFFFFFF'` mask and let natural overflow occur via a modular cast.

**Database migration** — Replace `generate_wallet_id_from_phone`:
- Change `h1 := h1 & x'7FFFFFFF'::int;` to allow signed overflow (use a bigint intermediate and cast back to int to simulate JS `| 0`)
- Same for `h2`

This is a single migration. No JS changes needed — the JS version is the "source of truth" since it's what users see.
